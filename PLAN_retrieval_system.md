# Hybrid Retrieval System - Implementation Plan

## 1. Overview

Build a multi-stage hybrid retrieval system over the existing `reel_analysis` data. The Instagram reel ingestion pipeline (download + analyze) feeds data in. This system sits on top: generating embeddings/chunks/concepts in a batch LangGraph workflow, and exposing search + typeahead via FastAPI endpoints.

**Data flow:**
```
Reel Ingestion (existing) → reel_analysis table (lifecycle_state: ANALYZED)
                                      ↓
                            Embedding Batch Job (new LangGraph graph)
                                      ↓
                            reel_analysis + embeddings tables populated (lifecycle_state: EMBEDDED)
                                      ↓
                            Search + Typeahead endpoints (FastAPI)
```

| Search output | Matched chunks with parent document context |
| Embedding pipeline | Separate batch LangGraph workflow |

| Data mapping | 1 reel = 1 document |

## 3. Database Schema

### 3.1 New Tables

#### `reel_documents`
Main retrieval document table, one row per reel.

```sql
CREATE TABLE reel_documents (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    reel_analysis_id TEXT NOT NULL REFERENCES reel_analysis(message_id),
    title TEXT NOT NULL,                              -- Generated from summary via Gemini
    summary TEXT NOT NULL,                            -- From summary_json.summary
    tags JSONB NOT NULL DEFAULT '[]',                 -- From summary_json.tags (existing hashtag-style)
    concepts JSONB NOT NULL DEFAULT '[]',             -- Generated by Gemini during embedding
    content TEXT NOT NULL,                            -- transcript + visual_text concatenated
    summary_embedding VECTOR(1024),                   -- BGE-M3 dense embedding of summary
    summary_colbert_tokens JSONB,                     -- BGE-M3 ColBERT token embeddings for summary
    tags_colbert_tokens JSONB,                        -- BGE-M3 ColBERT token embeddings for tags
    concepts_colbert_tokens JSONB,                    -- BGE-M3 ColBERT token embeddings for concepts
    embedding_status TEXT NOT NULL DEFAULT 'PENDING', -- PENDING | COMPLETED | FAILED
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE(reel_analysis_id)
);
```

#### `document_chunks`
Chunk-level embeddings for content retrieval.

```sql
CREATE TABLE document_chunks (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    document_id UUID NOT NULL REFERENCES reel_documents(id) ON DELETE CASCADE,
    chunk_index INT NOT NULL,                         -- Ordinal position of this chunk
    chunk_text TEXT NOT NULL,                         -- The chunk content
    chunk_embedding VECTOR(1024),                     -- BGE-M3 dense embedding of chunk
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE(document_id, chunk_index)
);
```

### 3.2 Indexes

```sql
-- pgvector HNSW indexes for dense vector search
CREATE INDEX idx_reel_documents_summary_emb ON reel_documents
    USING hnsw (summary_embedding vector_cosine_ops);

CREATE INDEX idx_document_chunks_emb ON document_chunks
    USING hnsw (chunk_embedding vector_cosine_ops);

-- GIN indexes for JSONB tag/concept lookups (typeahead)
CREATE INDEX idx_reel_documents_tags ON reel_documents USING gin (tags);
CREATE INDEX idx_reel_documents_concepts ON reel_documents USING gin (concepts);

-- Status filter for batch job
CREATE INDEX idx_reel_documents_status ON reel_documents (embedding_status);
```

## 4. Embedding Generation Pipeline (LangGraph Workflow)

### 4.1 Graph: `reel-embedder`

A new LangGraph graph registered in `graphs.json` with the following nodes:

```
read_pending → generate_title_and_concepts → generate_summary_embeddings
     → chunk_content → generate_chunk_embeddings → generate_colbert_tokens
     → persist_embeddings → report
```

### 4.2 State Definition

File: `agent_framework/graphs/reel_embedder/state.py`

```python
class ReelEmbedderState(AgentState):
    # Input
    batch_size: int = 50
    # Processing
    pending_records: list[dict] = []
    processed_count: int = 0
    failed_count: int = 0
    errors: list[str] = []
```

### 4.3 Node Definitions

All nodes in: `agent_framework/graphs/reel_embedder/nodes/`

#### Node 1: `read_pending`
- Query `reel_analysis` for records with `lifecycle_state = 'ANALYZED'` and no corresponding `reel_documents` entry (or `embedding_status = 'PENDING'`)
- Parse `summary_json` to extract summary, tags, transcript, visual_text
- Return list of pending records (up to `batch_size`)

#### Node 2: `generate_title_and_concepts`
- For each pending record, call Gemini with a structured prompt to:
  - Generate a concise title (5-10 words) from the summary
  - Generate 5-8 semantic concepts (e.g., "database initialization", "oracle provisioning")
- Use the existing worker_pool for concurrent LLM calls with throttling
- Update records with generated title and concepts
- **Gemini prompt structure:**
  ```
  Given this summary of a video, generate:
  1. A concise title (5-10 words)
  2. 5-8 semantic concepts that capture the search intent

  Summary: {summary}
  Tags: {tags}

  Return JSON: {"title": "...", "concepts": ["...", ...]}
  ```

#### Node 3: `generate_summary_embeddings`
- For each record, call LM Studio BGE-M3 endpoint to generate dense embedding of the summary
- LM Studio exposes an OpenAI-compatible API at a configurable base URL
- Store `summary_embedding` (1024-dim vector)
- Use worker_pool for concurrent calls

#### Node 4: `chunk_content`
- Concatenate `transcript` + `visual_text` into content string
- Apply LangChain `RecursiveCharacterTextSplitter`:
  - `chunk_size`: 512 characters
  - `chunk_overlap`: 100 characters
  - `separators`: ["\n\n", "\n", ". ", " ", ""]
- Store chunks with their `chunk_index`

#### Node 5: `generate_chunk_embeddings`
- Batch all chunks across documents
- Call LM Studio BGE-M3 endpoint to generate dense embeddings for each chunk
- Store `chunk_embedding` (1024-dim vector) per chunk

#### Node 6: `generate_colbert_tokens`
- For each document, call LM Studio BGE-M3 endpoint in ColBERT mode to generate token-level embeddings for:
  - Summary text → `summary_colbert_tokens`
  - Tags joined by space → `tags_colbert_tokens`
  - Concepts joined by space → `concepts_colbert_tokens`
- Each ColBERT output is a list of token vectors (each 1024-dim)
- Stored as JSONB: `[[0.1, 0.2, ...], [0.3, 0.4, ...], ...]`

#### Node 7: `persist_embeddings`
- Insert/upsert `reel_documents` rows with all generated data
- Insert `document_chunks` rows with chunk embeddings
- Update `reel_analysis.lifecycle_state` to `'EMBEDDED'` for processed records
- Update `reel_documents.embedding_status` to `'COMPLETED'`
- Wrap in a database transaction per document for atomicity

#### Node 8: `report`
- Log summary: processed count, failed count, errors
- Return final state

### 4.4 Edge Conditions

File: `agent_framework/graphs/reel_embedder/edges/conditions.py`

- `has_pending_records(state)` → continue to generate_title_and_concepts, else skip to report
- `has_failures(state)` → log errors but continue processing remaining records

## 5. LM Studio BGE-M3 Integration

### 5.1 Connection

LM Studio exposes an OpenAI-compatible embeddings API. Configuration via environment variables:

```
LM_STUDIO_BASE_URL=http://localhost:1234/v1
LM_STUDIO_MODEL=BAAI/bge-m3
LM_STUDIO_EMBEDDING_DIM=1024
```

### 5.2 Embedding Client

File: `app/services/embedding_client.py`

Two backends for different embedding types:

**Dense embeddings (LM Studio):**
- Uses `langchain_openai.OpenAIEmbeddings` pointed at LM Studio base URL
- Methods:
  - `embed_texts(texts: list[str]) -> list[list[float]]` - Dense embeddings for summaries/chunks
  - `embed_query(text: str) -> list[float]` - Query embedding for search

**ColBERT token embeddings (FlagEmbedding, in-process):**
- Uses `FlagEmbedding.BGEM3FlagModel` loaded at startup
- Methods:
  - `embed_colbert_tokens(text: str) -> list[list[float]]` - Per-token embeddings for ColBERT
  - `embed_colbert_tokens_batch(texts: list[str]) -> list[list[list[float]]]` - Batch version
- Model loaded once, reused across calls (~2GB memory)

### 5.3 BGE-M3 ColBERT Mode

LM Studio is used for dense embeddings only. For ColBERT token-level embeddings, use the **FlagEmbedding Python library directly** in the application process:

- Install: `pip install FlagEmbedding`
- Load model: `from FlagEmbedding import BGEM3FlagModel`
- Call: `model.encode(texts, return_dense=False, return_colbert_vecs=True)`
- Returns per-token embeddings (each token → 1024-dim vector)
- Model is loaded once at startup and reused across calls
- Memory requirement: ~2GB for BGE-M3 model weights

### 6.2 ColBERT MaxSim Implementation

File: `app/services/colbert_search.py`

```python
import numpy as np

def maxsim_score(query_tokens: np.ndarray, doc_tokens: np.ndarray) -> float:
    """
    Compute ColBERT MaxSim score.
    query_tokens: shape (Q, D) - query token embeddings
    doc_tokens: shape (S, D) - document token embeddings
    Returns: scalar score (higher = more relevant)
    """
    # Similarity matrix: (Q, S)
    sim_matrix = query_tokens @ doc_tokens.T
    # MaxSim: for each query token, take max similarity across all doc tokens
    max_sims = sim_matrix.max(axis=1)
    # Sum across query tokens
    return float(max_sims.sum())
```

For efficiency with ~1K-10K docs:
- Load candidate document token embeddings in batches from DB
- Compute MaxSim using numpy vectorized operations
- This is feasible at this scale since queries are infrequent

### 6.5 Full Search Flow

```python
async def search(query: str, top_n: int = 20) -> list[SearchResult]:
    # 1. Retrieve candidates from all 5 stages
    candidates = await retrieve_candidates(query, top_k=100)

    # 2. Rerank with hosted API
    ranked = await rerank(query, candidates, top_n=top_n)

    # 3. Enrich results with matched chunks and parent document context
    results = []
    for item in ranked:
        doc = await fetch_document(item.document_id)
        matched_chunks = await fetch_top_chunks(item.document_id, item.chunk_ids)
        results.append(SearchResult(
            document=doc,            # title, summary, tags, concepts
            matched_chunks=matched_chunks,  # top chunks with scores
            relevance_score=item.score
        ))

    return results
```
